# Persisted task lifecycle

The `vela-kernel` crate provides a deliberately small task lifecycle: starting, recording typed execution observations, completing with an output, cancelling with a reason, failing with a diagnostic, associating with a session, and loading or listing tasks over the typed SQLite event log.

## Observable contract

- `TaskId` is an opaque, non-empty UTF-8 string. The store isolates task streams with the internal `task:` prefix.
- `TaskGoal` is a non-empty UTF-8 string. Whitespace is meaningful; the kernel does not trim or reinterpret caller input.
- `TaskOutput` is a non-empty UTF-8 completion output. Whitespace is meaningful; the kernel persists and returns the value without trimming or reinterpretation.
- `TaskCancellation` is a non-empty UTF-8 cancellation reason. Whitespace is meaningful; the kernel persists and returns the value without trimming or reinterpretation.
- `TaskFailure` is a non-empty UTF-8 diagnostic. Whitespace is meaningful; the kernel persists and returns the value without trimming or reinterpretation.
- `TaskObservationId` and `TaskObservationText` are caller-supplied opaque UTF-8 strings that must contain at least one non-whitespace character. Surrounding whitespace remains meaningful and is preserved. Observation IDs are unique within a task.
- `TaskObservationKind` has exactly four persisted values: `attempt`, `diagnostic`, `correction`, and `verification`. `TaskStore::append_observation` appends ungrouped evidence, while `TaskStore::append_observation_for_attempt` relates a diagnostic, correction, or verification to one earlier attempt in the same task. Both append one `task.observation_appended` event at payload version `2`; the returned and replayed task exposes observations in task stream order and each observation's optional parent attempt ID.
- The task aggregate owns observation ordering and parent validation; there is no separate execution-episode aggregate. Attempt observations cannot have parents. A supplied parent must already exist in the same task and have kind `attempt`. Missing parents, non-attempt parents, and parented attempts return typed domain errors without writing. Ungrouped observations remain supported.
- Observation payload-version-1 events remain replayable as ungrouped evidence without migration or stream rewriting. Replay enforces the same parent rules for payload-version-2 events, so missing, non-attempt, forward, and parented-attempt relations are invalid history.
- Task observations are append-only execution evidence. Completion, cancellation, or failure freezes the observation history; appending afterward returns the task's existing terminal-state error without changing the stream. Missing tasks return `NotFound`, and a reused observation ID returns `DuplicateObservation`, also without writing.
- Replayed observation payloads are validated with the same non-blank rules. Unsupported kinds or payload versions and malformed values surface through `TaskStoreError::Replay`; duplicate IDs or observations after a terminal event are invalid history rather than silently reinterpreted evidence.
- A task starts without a session association. `TaskStore::associate_session` requires an existing task and an existing open session, then appends one `task.session_associated` event at payload version `1`. Its payload carries both identifiers, and replay projects the session identifier into the task. The append atomically guards both the task version and the observed session version, so a racing session close either follows a valid association or causes association to reload and reject the closed session.
- Association is immutable. Repeating it with either the same or another target returns `TaskStoreError::AlreadyAssociated` with the persisted session identifier and appends no event. A missing target returns `SessionNotFound`, and a closed target returns `SessionClosed`, also without appending.
- Association is independent of task terminal status: active and terminal tasks may be associated, and a later task terminal transition preserves an existing association. Closing or reopening the session does not alter the task stream or projected association.
- `TaskStore::start` appends one `task.started` event at payload version `1` with `ExpectedVersion::NoStream`, then returns a task whose status is `Active`.
- `TaskStore::complete` loads an existing active task and appends a `task.completed` event containing the supplied output at payload version `2` with the exact current stream version. It returns the task with `Completed` status and the same output.
- `TaskStore::cancel` loads an existing active task and appends a `task.cancelled` event containing the supplied reason at payload version `2` with the exact current stream version. It returns the task with `Cancelled` status and the same reason.
- `TaskStore::fail` loads an existing active task and appends a `task.failed` event containing the supplied diagnostic at payload version `2` with the exact current stream version. It returns the task with `Failed` status and the same diagnostic.
- `TaskStore::load` replays the task stream. It returns the same ID, goal, current status, terminal detail, optional session association, and ordered observations after reopening the database. A missing stream returns `None`.
- `TaskStore::list` discovers task streams from their existing start events, replays every task from one consistent SQLite read snapshot, and returns them in ascending `TaskId` order. Each task includes its latest lifecycle status, terminal detail, optional session association, and ordered observations. An empty store returns an empty list, session streams remain excluded even when their external identifiers match task identifiers, and reopening the database does not change the result. Listing appends no events and does not maintain a separate mutable index. A malformed owning task stream identifier returns `TaskStoreError::InvalidStreamId`.
- `TaskStore::list_for_session` requires an existing session and returns every task associated with it in ascending `TaskId` order. Each result is replayed to its latest state, so active and terminal tasks are included. Closed and reopened sessions retain their membership, tasks associated with other sessions and unassociated tasks are excluded, and reopening the database does not change the result. A missing session returns `SessionNotFound` and the query appends no events.
- Newly completed tasks expose the output persisted by their completion event. Legacy payload-version-1 completion events with the old empty payload remain replayable and expose no output. Active, cancelled, and failed tasks expose no completion output.
- Newly cancelled tasks expose the reason persisted by their cancellation event. Legacy payload-version-1 cancellation events with the old empty payload remain replayable and expose no reason. Active, completed, and failed tasks expose no cancellation reason.
- Active, completed, and cancelled tasks expose no failure diagnostic. Newly failed tasks expose the diagnostic persisted by their failure event. Legacy payload-version-1 failure events with the old empty payload remain replayable and expose no diagnostic.
- Starting an existing ID returns `TaskStoreError::AlreadyExists` and leaves the original history unchanged. The event log's expected-version transaction also enforces this under racing writers.
- Completing a missing ID returns `TaskStoreError::NotFound` without creating a stream. Completing an already completed task returns `TaskStoreError::AlreadyCompleted` and leaves the completed history unchanged; exact-version appends also enforce the transition under racing writers.
- Cancelling a missing ID returns `TaskStoreError::NotFound` without creating a stream. A repeated cancellation returns `TaskStoreError::AlreadyCancelled`.
- Failing a missing ID returns `TaskStoreError::NotFound` without creating a stream. A repeated failure returns `TaskStoreError::AlreadyFailed`.
- Every terminal transition rejects a task already completed, cancelled, or failed with the error for its persisted status and preserves the winning history.
- Terminal transitions racing on the same active task persist exactly one terminal event. The losing writer reloads the winning event and reports the persisted terminal state.
- Unknown event discriminators or payload versions and malformed payloads, including an empty persisted completion output, cancellation reason, failure diagnostic, or association identifier, remain explicit `ReplayError` values wrapped by `TaskStoreError::Replay`; persisted data is never skipped.
- Valid histories start with one `task.started` event, contain uniquely identified observation events only while active, contain at most one `task.session_associated` event, and contain at most one terminal `task.completed`, `task.cancelled`, or `task.failed` event. Association may appear before or after the terminal event. Terminal-first, duplicate starts, observation IDs, associations, or terminal events, observation events after a terminal event, other events after a terminal event besides the single association, and association payloads naming a different task are invalid history rather than implicit state changes.
- `AssistantRuntime::execute_task_turn` can append an `Attempt` observation after a successful assistant turn for an active associated task. `AssistantRuntime::execute_task_correction_turn` can instead preflight an existing parent attempt and append the successful response as a parented `Correction`. Callers own both new observation IDs. Transcript commits precede the authoritative task append, so a racing task-store failure preserves the completed conversation turn; known duplicate, missing-parent, and non-attempt-parent correction requests fail during preflight. See [`assistant-runtime.md`](assistant-runtime.md) for the complete partial-commit contract.

`TaskStoreError` is non-exhaustive. Wrapped event-log, replay, and session-store failures are exposed through `std::error::Error::source`; domain errors such as `AlreadyExists`, `NotFound`, `AlreadyCompleted`, `AlreadyCancelled`, `AlreadyFailed`, `SessionNotFound`, `SessionClosed`, `AlreadyAssociated`, `DuplicateObservation`, `AttemptCannotHaveParent`, `ParentObservationNotFound`, `ParentObservationNotAttempt`, `InvalidStreamId`, and `InvalidHistory` have no source.

## Non-goals

This slice does not add a separate execution-episode aggregate or lifecycle, multiple or cross-task parents, generic runtime evidence classification, runtime diagnostics or verifications, a structured result, failure, or cancellation taxonomy, artifacts, stack traces, retry policy, timestamps, actors, model messages, tools, child tasks, moving or detaching associations, a duplicate mutable session-side task index, pagination, async execution, cooperative runtime cancellation, automatic lifecycle transitions, or cross-stream runtime transactions. Those require separate lifecycle events and acceptance tests rather than assumptions in persisted state.

See [`event-log.md`](event-log.md) for the underlying append, durability, concurrency, and replay guarantees.
